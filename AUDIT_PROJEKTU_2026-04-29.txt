1. SHRNUTÍ (TL;DR)
════════════════════════════════════════════════════════════════════════════════

Projekt má SOLIDNÍ základ — autentizace, CSRF, prepared statements, escaping,
rate limiting, 2FA. Většina věcí je udělaná správně.

Hlavní rezervy: bezpečnost při deploy (HTTPS, retention dat), pozorování
chyb (logging) a maintenance velkých view souborů.

Z 16 nálezů auditu jsou DNES opravené 4 kritické (#1, #3, #4, #5).
Zbylých 12 je popsaných níž — žádný není akutní, ale postupně by se
měly řešit. Jeden (HTTPS) je VÍCE NEŽ AKUTNÍ ale počká až bude server.

2. CO BYLO OPRAVENO (29. 4. 2026)

──────────────────────────────────────────────────────────────────
FIX #1: Šifrování AES-256-CBC -> AES-256-GCM
──────────────────────────────────────────────────────────────────
PŮVODNÍ STAV:  CBC bez HMAC = zranitelné na padding oracle attack.
NOVÝ STAV:     GCM (autentizovaná šifra), prefix "v2:", IV 12 B + tag 16 B.
                Stará data v DB jsou DÁL ČITELNÁ — decrypt rozpozná
                "v2:" prefix (nový format) i bez něj (starý CBC).
                Encrypt vždy produkuje nový GCM formát.
SOUBOR:        config/constants.php (ř. 107-205)
DOPAD:         Žádný pro běžící data. Po deployi se nová zašifrovaná
                data postupně migrují (přepis na GCM při edit).


──────────────────────────────────────────────────────────────────
FIX #3: Tichá PDOException -> logované
──────────────────────────────────────────────────────────────────
PŮVODNÍ STAV:  52x prázdné `catch (\PDOException) {}` v 3 controllerech.
                Když DB selže, frontend ukáže prázdná data bez stopy.
NOVÝ STAV:     Helper `crm_db_log_error($e, $context)` v helpers/audit.php.
                35 prázdných catch bloků nahrazeno za logované —
                používá `error_log()` (jde do PHP error logu / Apache
                error logu / kamkoli máš nakonfigurované).
SOUBOR:        app/helpers/audit.php (nový helper)
                app/controllers/OzController.php (require + 26 catchů)
                app/controllers/BackofficeController.php (require + 8 catchů)
                app/controllers/CallerController.php (1 catch)
DOPAD:         Žádný pro happy-path. Pokud něco selže, uvidíš:
                "[CRM DB] [OzController::getLeads] Table x doesn't exist..."

POZNÁMKA:      18 catchů s VYSVĚTLUJÍCÍMI komentáři a fallback hodnotami
                (např. "tabulka ještě neexistuje, necháváme prázdné")
                jsem nesahal — nejsou tichá ztráta, jsou to dokumentované
                graceful degradation. Viz bod 3.13 v sekci ODLOŽENO.


──────────────────────────────────────────────────────────────────
FIX #4: CSV retention v storage/imports/
──────────────────────────────────────────────────────────────────
PŮVODNÍ STAV:  Naimportované CSV s telefony/IČO/emaily zůstávaly
                na disku věčně. Při kompromisu FTP / shellu = export DB.
NOVÝ STAV:     Skript `tools/cleanup_imports.php` — smaže imp_*.csv

                Plus `storage/.htaccess` se "Require all denied"

SOUBORY:       tools/cleanup_imports.php (nový)

3.1  HTTPS — VYNUCENÍ PO NASAZENÍ NA PRODUKČNÍ SERVER  [VYSOKÁ PRIORITA]
────────────────────────────────────────────────────────────────────────────────

KDE:    public/.htaccess (řádky 11-14)

PROBLÉM:
  HTTPS redirect je zakomentovaný:
    # RewriteCond %{HTTPS} !=on
    # RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

  Bez tohohle:
  - Útočník na stejné WiFi vidí session cookies
  - "Secure" flag na cookies pomůže jen po HTTPS-only přístupu;
     před první HTTPS návštěvou jsou v cleartextu

CO UDĚLAT (po nasazení certifikátu na produkci):
  1. V public/.htaccess odkomentuj řádky 11-14 (jen na produkci!)
  2. Přidat Strict-Transport-Security header (HSTS):
       Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
  3. Nastavit cookie SameSite na "Strict" (teď je "Lax") v session.php

DOPORUČENÝ HOSTING:
  - Pokud chceš Czech-friendly: WEDOS, Forpsi (sdílený s Let's Encrypt)
  - Pokud spíš VPS: DigitalOcean / Linode (4GB RAM stačí), Apache + MariaDB
  - Vyhni se shared hostingu kde nemáš SSH přístup — nebudeš moct cron,
     dobře audit logy, potřebné PHP rozšíření atd.

  Až budeš ready, dej vědět specs serveru — pomůžu s konfigurací.

3.3  TABULKY V CONTROLLERECH PŘES ensureXTable()  [STŘEDNÍ]
────────────────────────────────────────────────────────────────────────────────

KDE:    OzController::ensureWorkflowTable(), ensureNotesTable() a další

PROBLÉM:
  Některé controllery vytvářejí tabulky ZA BĚHU pomocí
  CREATE TABLE IF NOT EXISTS při každém requestu. To je:
  - Zbytečný overhead na každý GET/POST
  - Race condition při paralelních requestech
  - Schránka schema změn v controllerech místo migrací

CO UDĚLAT:
  Po vyřešení 3.2 (migrační framework):
  - Přesunout schema z ensureXTable() do nových migrací
  - V controllerech ensureXTable() už nevolat
  - Po deploy spustit `php tools/migrate.php`

ČAS:    1-2 hodiny po vyřešení 3.2.

3.5  ŽÁDNÝ CSP HEADER (Content-Security-Policy)  [STŘEDNÍ]
────────────────────────────────────────────────────────────────────────────────

PROBLÉM:
  Máš X-Frame-Options, X-Content-Type-Options, Referrer-Policy,
  ale CSP chybí. CSP by zúžil útočnou plochu pro XSS — i kdyby
  se útočníkovi podařilo injectovat skript, browser ho neprovede,
  pokud není z povolené origin.

CO UDĚLAT:
  Začni s mírnou polikou (kvůli velkému množství inline JS):

    Header always set Content-Security-Policy "
      default-src 'self';
      script-src 'self' 'unsafe-inline';
      style-src 'self' 'unsafe-inline';
      img-src 'self' data:;
      connect-src 'self';
      font-src 'self' data:;
      frame-ancestors 'self';
      form-action 'self';
    "

  POTOM postupně refactorovat inline JS do externích souborů
  a odebírat 'unsafe-inline' (nahradit nonces).

ČAS:    30 min počáteční konfigurace + průběžně.

3.7  .env SOUBOR V REPU  [VYSOKÁ — POKUD JE V GITU]
────────────────────────────────────────────────────────────────────────────────

PROBLÉM:
  V project rootu je `.env` soubor. Pokud je commitnutý v gitu,
  máš tam tajné údaje (CRM_ENCRYPTION_KEY, DB hesla, SMTP creds...)
  veřejně dostupné.

CO UDĚLAT:
  1. Zkontroluj `.gitignore`:
     cat .gitignore | grep -E "\.env"

     Mělo by tam být:
       .env
       .env.local
       .env.*.local

  2. Pokud .env BYL commitnutý v historii:
     - Změň všechny credentials (DB heslo, encryption key, SMTP, SMS)
     - Použij `git filter-repo` nebo BFG Repo-Cleaner pro vyčištění historie
     - Push s --force (ALE BACKUPNI BEFORE!)

  3. Vytvoř `.env.example` se vzorovými klíči (bez hodnot):
     CRM_APP_URL=https://your-domain.example
     CRM_ENCRYPTION_KEY=replace-with-64-hex-chars
     ...

ČAS:    20 min kontrola, 1-2 h pokud se musí čistit historie.

3.8  JSON V INLINE onclick ATRIBUTECH  [NÍZKÁ]
────────────────────────────────────────────────────────────────────────────────

KDE:    oz/leads.php — tlačítka "Předat BO" (právě opravené přechodem
        z onclick="..." na onclick='...')

PROBLÉM:
  Pattern `onclick="...<?= json_encode($x) ?>..."` se může v budoucnu
  snadno znovu rozbít. Vnější uvozovky HTML atributu (`"`) kolidují
  s vnitřními JSON uvozovkami (`"`). Pokud někdo přidá další podobné
  tlačítko a nevšimne si, že má použít single-quoted atribut, dostane
  silently broken click.

CO UDĚLAT:
  Refactor na data attributes:

    <button data-bo-cid="123"
            data-bo-title="..."
            data-bo-body="..."
            onclick="ozPredatBoConfirmFromBtn(this)">
      Předat BO
    </button>

  A v JS:
    function ozPredatBoConfirmFromBtn(btn) {
      const cId   = parseInt(btn.dataset.boCid, 10);
      const title = btn.dataset.boTitle;
      const body  = btn.dataset.boBody;
      ozPredatBoConfirm(cId, title, body);
    }

  Tím odpadají problémy s escapováním — HTML atributy escapují
  bezpečně přes htmlspecialchars / crm_h.

ČAS:    30 min na 2 tlačítka.

3.13 ZBYLÉ "GRACEFUL FALLBACK" PDOEXCEPTION CATCHES  [NÍZKÁ]
────────────────────────────────────────────────────────────────────────────────

PROBLÉM:
  Po dnešní opravě (#3) zůstalo 18 catch bloků s vysvětlujícími
  komentáři ("tabulka neexistuje, necháváme prázdné") a explicitní
  fallback hodnotou. Tyhle nejsou tichá ztráta — jsou to dokumentované
  graceful degradation. Ale stále by stálo za to je logovat —
  pokud by tabulka znenadání zmizela, nebo dotaz selhal z jiného
  důvodu, nedokážeme to zjistit.

CO UDĚLAT:
  V dotčených catch blocích doplnit:

    } catch (\PDOException $e) {
        crm_db_log_error($e, __METHOD__);
        $tabCounts['chybne_oz'] = 0;  // existující fallback zůstává
    }

ČAS:    20 min na všech 18 výskytů.
